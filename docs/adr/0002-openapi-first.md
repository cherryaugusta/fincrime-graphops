# ADR 0002: OpenAPI-First API Contract for FinCrime GraphOps Day Zero

- Status: Accepted
- Date: 2026-03-08
- Deciders: Project Maintainer
- Technical Story: Establish a typed, contract-driven API baseline between the Django backend and Angular client

## Context

FinCrime GraphOps is intended to demonstrate disciplined engineering practices in a sensitive problem domain involving sanctions, beneficial ownership, graph-linked entities, and AI governance. Even at Day Zero, the project should avoid loosely defined backend-to-frontend integration.

The initial architecture includes:

- a Django REST API
- an Angular strict-mode client
- PostgreSQL and Redis as backend dependencies
- Swagger UI and machine-readable schema exposure through drf-spectacular

Without an explicit API contract, several risks emerge early:

- backend and frontend models drift independently
- client code makes assumptions that are not enforced
- endpoint behavior becomes implicit rather than documented
- later feature growth becomes harder to govern and test
- generated or typed client code becomes difficult to introduce cleanly

Because this project is intended as a portfolio-grade demonstration, the API surface should be described explicitly and exposed in a standard format from the start.

## Decision

We will adopt an OpenAPI-first approach from Day Zero.

This means the HTTP contract is treated as a first-class project artifact rather than an incidental by-product of implementation. The backend must expose a machine-readable schema and interactive documentation, and the frontend must consume backend data through explicitly typed contracts aligned with the schema.

The Day Zero implementation includes the following decisions.

### 1. The backend must expose an OpenAPI schema

The Django API will expose an OpenAPI schema endpoint at:

- `/api/schema/`

This schema is generated by `drf-spectacular` and configured through Django REST Framework settings.

### 2. The backend must expose interactive API documentation

The Django API will expose Swagger UI at:

- `/api/docs/`

This provides a browsable representation of the current contract and supports local inspection, onboarding, and manual verification.

### 3. Django REST Framework must use a single default schema mechanism

The API will standardize on:

- `drf_spectacular.openapi.AutoSchema`

This ensures schema generation is consistent and centrally configured.

### 4. The Angular client must use explicit typed contracts

The frontend must define TypeScript interfaces representing the shapes it consumes from backend responses.

At Day Zero this includes:

- `Entity`
- `DependencyStatus`
- `HealthResponse`

These contracts live in:

- `client/src/app/contracts/entity.ts`

This keeps client assumptions explicit and reviewable.

### 5. API access in the frontend must be mediated by a dedicated typed service

The Angular client must call backend endpoints through a dedicated service rather than scattering raw HTTP calls throughout components.

At Day Zero this is implemented in:

- `client/src/app/contracts/typed-api.service.ts`

This makes contract evolution easier, centralizes HTTP concerns, and improves maintainability.

### 6. The health endpoint is the first contract-verified path

The initial backend-to-frontend contract will be demonstrated with the health endpoint:

- `GET /api/health/`

This endpoint returns a structured JSON response containing:

- service identity
- overall health status
- dependency-level database status
- dependency-level Redis status
- correlation ID

This is sufficient to prove the full path:

- backend implementation
- schema exposure
- frontend typed consumption
- local cross-origin integration
- traceability support

### 7. Contract evolution must prefer additive change

As the project grows, API changes should prefer additive evolution over breaking changes where practical.

Examples:

- add new fields rather than repurpose existing ones
- add new endpoints rather than silently change established semantics
- deprecate intentionally rather than remove abruptly

This reduces churn in the frontend and supports controlled growth.

### 8. Contract documentation must remain aligned with implementation

The schema and Swagger docs are not decorative artifacts. They are part of the working project baseline.

A change to endpoint behavior, serializer shape, or expected request/response format should be reflected in the exposed schema and reviewed as part of normal development.

### 9. Typed integration is preferred over implicit JSON handling

The project will avoid generic untyped handling of backend responses in the frontend where a typed model is practical.

This improves:

- readability
- refactor safety
- editor tooling
- defect detection
- portfolio clarity

## Consequences

### Positive consequences

- backend and frontend integration becomes more explicit
- API behavior is easier to inspect and reason about
- frontend models are clearer and safer under strict TypeScript settings
- onboarding improves because API docs are visible immediately
- future code generation or schema-based validation is easier to introduce
- the project better demonstrates mature API engineering practice

### Negative consequences

- maintaining schema alignment adds some development overhead
- careless manual TypeScript contracts may still drift if not updated alongside backend changes
- small experimental changes may feel slower because they should be documented and typed
- OpenAPI generation adds another dependency and configuration surface

### Trade-offs accepted

- we accept a small amount of setup and maintenance overhead in exchange for clearer contracts
- we accept manual typed contracts at Day Zero rather than adding full client generation immediately
- we accept that the first contract surface is small because proving the workflow matters more than early breadth

## Alternatives considered

### 1. Code-first only, with no contract emphasis

Rejected.

Although the backend implementation naturally produces behavior, relying only on code and informal knowledge would weaken frontend safety and reduce the clarity of the repository as a portfolio artifact.

### 2. Frontend consumes raw JSON without explicit interfaces

Rejected.

This would be faster initially but would undermine strict typing, make integration drift more likely, and reduce maintainability.

### 3. Generate the frontend client automatically on Day Zero

Not selected for Day Zero.

Automatic client generation from OpenAPI is a valid future option, but it adds complexity that is not necessary for the initial baseline. Day Zero uses manual typed contracts to keep the setup understandable while preserving contract discipline.

### 4. Document endpoints only in README text

Rejected.

README-only API documentation is insufficient because it is not machine-readable, can drift easily, and does not support tooling or interactive exploration.

## Implementation notes

The Day Zero implementation of this ADR is reflected in:

- `api/config/settings.py`
- `api/config/urls.py`
- `api/core/serializers.py`
- `api/core/views.py`
- `api/core/urls.py`
- `client/src/app/contracts/entity.ts`
- `client/src/app/contracts/typed-api.service.ts`
- `client/src/app/app.ts`
- `client/src/main.ts`
- `README.md`

Key implementation details include:

- DRF uses `DEFAULT_SCHEMA_CLASS = "drf_spectacular.openapi.AutoSchema"`
- drf-spectacular exposes `/api/schema/`
- Swagger UI is served at `/api/docs/`
- Angular uses `provideHttpClient()`
- the client calls the backend through a typed service
- the health response is rendered in the root Angular app for Day Zero verification

## Compliance and review notes

This ADR governs engineering approach, not regulatory compliance. OpenAPI-first design improves clarity, integration discipline, and reviewability, but it does not by itself make the system safe for production or regulated use.

This ADR should be revisited when any of the following occur:

- request bodies are added for create or update endpoints
- authentication and authorization are introduced
- schema versioning becomes necessary
- generated clients are introduced
- external consumers beyond the Angular client begin using the API
- breaking contract changes become likely
